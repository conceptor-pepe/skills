# Grill Me（设计压力测试）

对用户的设计或方案进行无死角追问，走完决策树的每个分叉，直到我们对方案形成完整共识。

## 执行原则

- **每次只问一个问题** — 不要一次抛出多个问题
- **先探索再问** — 能通过查看代码库回答的问题，先查再问
- **沿决策树推进** — 一个问题解决后，立即推进下一个分叉
- **不接受模糊答案** — 如果回答不够明确，继续追问直到清晰
- **覆盖异常路径** — 不只追问主路径，也追问边界、失败、并发场景

## 追问维度（参考，不局限于此）

**功能完整性**
- 这个方案覆盖了哪些用户故事？遗漏了哪些？
- 边界情况如何处理？（空值、超限、并发写入）
- 用户操作失败时的体验是什么？

**技术可行性**
- 这个方案依赖哪些现有基础设施？
- 有哪些技术风险或未验证的假设？
- 性能瓶颈在哪里？预期数据量级是什么？

**接口设计**
- API 的幂等性如何保证？
- 向后兼容性如何处理？
- 鉴权/权限边界是什么？

**数据一致性**
- 哪些操作需要事务？
- 失败回滚策略是什么？
- 数据迁移如何处理？

**可测试性**
- 如何验证这个方案是正确的？
- 哪些部分难以测试？如何解决？

**可维护性**
- 这个设计六个月后还容易改吗？
- 有没有引入不必要的耦合？

## 结束条件

当以下条件都满足时，停止追问：

- [ ] 所有主路径的决策都已明确
- [ ] 所有关键边界情况都有处理策略
- [ ] 技术风险已被识别并有应对思路
- [ ] 没有明显的"我们还没想清楚"的分叉

## 输出决策记录

追问结束后，输出一份决策记录：

```markdown
## 设计决策记录

### 已确认的决策
- 决策 1：...（理由：...）
- 决策 2：...（理由：...）

### 识别的风险
- 风险 1：...（应对：...）

### 遗留问题（需要进一步确认）
- 问题 1：...
```

这份记录可以直接用作 PRD 的「实施决策」章节，或作为 `write-a-prd` 的输入。
