# Go 代码编写规范

在编写 Go 代码时，必须严格遵循以下规范：

## 文件职责原则
- **单一职责原则**：每个文件专注于单一概念
  - 例如：一个 handler、一个 service，或一组相关的工具函数
  - 避免在一个文件中混合多个不相关的概念

## 函数职责原则
- **单一职责原则**：每个函数只做一件事，并做好
  - 函数应该简短、专注、易于理解
  - 如果一个函数做了多件事，应该拆分为多个函数
  - 单个函数不要超过50行

## 命名规范
- **自解释性命名**：使用简洁、清晰的名称（1-3 个单词）
  - 避免冗余的命名（如 `UserUser`、`GetUserData` 应简化为 `GetUser`）
  - 变量名应该清楚地表达其用途
  - 函数名应该是动词或动词短语

## 注释规范
- **注释说明原因，而非内容**：
  - 不要注释显而易见的代码（如 `// 设置变量 x = 1`）
  - 应该注释为什么这样做（如 `// 使用缓存避免重复查询数据库`）
  - 复杂逻辑必须添加注释说明设计意图

- **函数注释规范**：
  - 所有公开函数（首字母大写）必须添加函数注释
  - 函数注释使用 Go doc 格式，以函数名开头
  - 注释应该简洁说明函数的功能和用途，不超过 2-3 行
  - 参数说明使用 `@param` 格式，每个参数一行，简洁明了（不超过 10 个字）
  - 返回值说明使用 `@return` 格式（如果有返回值）
  - 示例：
    ```go
    // GetUser 根据用户ID获取用户信息
    // @param ctx 上下文对象，用于控制请求超时和取消
    // @param id 用户ID
    // @return 用户对象和错误信息
    func GetUser(ctx context.Context, id int64) (*User, error)
    ```
  - 私有函数（首字母小写）如果逻辑复杂，也应该添加注释
  - 避免冗长的注释，优先通过清晰的命名和代码结构表达意图

## 代码复用原则
- **不重复造轮子**：
  - 优先使用标准库（stdlib）提供的功能
  - 优先使用项目中已有的工具函数和实用程序
  - 在创建新功能前，先检查是否有现成的解决方案

## 语法复杂度限制
- **避免高级语法特性**：
  - 不使用反射（reflection），除非绝对必要
  - 避免复杂的泛型（generics）使用
  - 不使用过于巧妙的技巧或 hack
  - **优先考虑可读性而非聪明性**

## 控制流规范
- **避免深层嵌套**：
  - if 语句嵌套深度不超过 2 层
  - 超过 2 层嵌套时，使用早期返回（early return）模式重构
  - 优先使用卫语句（guard clauses）减少嵌套
  - 示例：使用 `if condition { return }` 而不是 `if !condition { ... }`

## 错误处理规范
- **统一错误处理**：
  - 使用项目统一的错误类型（如 `pkg/errors`）
  - 错误信息应该清晰、可操作
  - 不要忽略错误，使用 `_` 忽略时必须注释原因
  - 错误包装时使用 `fmt.Errorf("context: %w", err)` 保留错误链
  - 对于非高频、非持续触发的错误分支，必须记录错误日志，便于后续问题定位
  - 示例：`return fmt.Errorf("查询用户失败: %w", err)` 而不是 `return err`

## 资源管理规范
- **及时释放资源**：
  - 使用 `defer` 确保资源释放（文件、连接、锁等）
  - `defer` 语句应该紧跟在资源获取之后
  - 使用 `context.WithTimeout` 或 `context.WithCancel` 管理超时
  - 避免资源泄漏，特别是 goroutine 和数据库连接
  - 示例：`defer file.Close()` 而不是在函数末尾手动关闭

## Context 使用规范
- **传递上下文**：
  - 所有可能阻塞的函数都应该接受 `context.Context` 作为第一个参数
  - 使用 `context.WithTimeout` 设置超时，避免无限等待
  - 不要将 context 存储在结构体中，应该作为参数传递
  - 示例：`func (s *Service) GetUser(ctx context.Context, id int64) (*User, error)`

## 函数长度和复杂度规范
- **函数应该简短**：
  - 函数长度不超过 50 行（理想情况下 20-30 行）
  - 超过 50 行的函数应该考虑拆分
  - 圈复杂度（Cyclomatic Complexity）不超过 10
  - 一个函数只做一件事，如果做了多件事应该拆分

- **函数参数个数限制**：
  - 函数参数个数建议不超过 5 个
  - 超过 5 个参数时，应该考虑使用结构体封装参数
  - 使用参数结构体可以提高可读性和可维护性，也便于后续扩展
  - 示例：
    ```go
    // 不推荐：参数过多
    func CreateUser(ctx context.Context, name string, email string, age int, phone string, address string) error
    // 推荐：使用结构体封装
    type CreateUserRequest struct {
        Name    string
        Email   string
        Age     int
        Phone   string
        Address string
    }
    func CreateUser(ctx context.Context, req *CreateUserRequest) error
    ```
  - 特殊情况（如配置对象、选项模式）可以适当放宽，但应优先考虑结构体封装

## 并发安全规范
- **资源竞争保护**：
  - 共享资源必须使用互斥锁或 channel 保护
  - 优先使用 channel 进行通信，而不是共享内存
  - 使用 `sync.Map` 处理高并发读多写少的场景
  - 避免在持有锁时调用可能阻塞的操作
  - 使用 `context.Context` 管理 goroutine 生命周期

## 接口设计规范
- **接口最小化原则**：
  - 接口应该小而专注，只包含必要的方法
  - 接口定义在消费方，而不是实现方
  - 避免过度抽象，只在真正需要多实现时定义接口
  - 示例：`type Reader interface { Read([]byte) (int, error) }` 而不是包含10个方法的大接口

## 日志规范
- **结构化日志**：
  - 使用结构化日志（如 zap），而不是格式化字符串
  - 日志级别要合理：Debug（调试）、Info（信息）、Warn（警告）、Error（错误）
  - 错误日志必须包含足够的上下文信息（请求ID、用户ID等）
  - 错误分支若不会持续高频打印，禁止静默返回，至少记录一条结构化错误日志
  - 避免在生产环境输出敏感信息（密码、token等）
  - 示例：`logger.Info("用户登录成功", zap.Int64("user_id", userID))`

## 变量作用域规范
- **最小作用域原则**：
  - 变量应该在最小必要的作用域内声明
  - 优先使用局部变量，避免不必要的包级变量
  - 使用 `:=` 进行短变量声明，而不是 `var`
  - 示例：`if err := doSomething(); err != nil { ... }` 而不是先声明再赋值

## 性能优化规范
- **避免不必要的分配**：
  - 预分配 slice 和 map 容量（使用 `make([]T, 0, capacity)`）
  - 避免在循环中重复分配大对象
  - 使用 `strings.Builder` 进行字符串拼接
  - 避免不必要的类型转换和接口转换
  - 只有在真正需要时才进行性能优化，优先保证代码可读性

## 代码审查检查清单
在提交代码前，确保：
- [ ] 每个文件只包含一个主要概念
- [ ] 每个函数只做一件事
- [ ] 所有命名都是自解释的
- [ ] 注释说明了"为什么"而非"是什么"
- [ ] 所有公开函数都有函数注释，包含参数和返回值说明
- [ ] 函数参数个数不超过 5 个，超过时使用结构体封装
- [ ] 没有重复实现标准库已有的功能
- [ ] 代码易于阅读和理解，没有过度复杂的语法
- [ ] if 语句嵌套深度不超过 2 层，使用早期返回模式减少嵌套
- [ ] 所有错误都被正确处理，没有忽略错误
- [ ] 非高频错误分支均已记录可定位日志
- [ ] 资源（文件、连接、锁）都使用 defer 正确释放
- [ ] 并发访问的资源都有适当的保护（锁或 channel）
- [ ] 函数长度不超过 50 行，复杂函数已拆分
- [ ] 使用了 context.Context 管理超时和取消
- [ ] 日志包含足够的上下文信息
- [ ] 接口设计遵循最小化原则
- [ ] 变量作用域最小化，优先使用局部变量
