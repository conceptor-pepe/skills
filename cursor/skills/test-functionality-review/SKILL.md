# 测试功能 Review（ST 黑盒测试专家）

> **核心理念**：本 SKILL 是**黑盒测试思维审计代码**，不是写测试代码。
> 我们站在"不知道内部实现"的角度设计测试用例，再用用例去审计代码，发现实现中的遗漏和 bug。

## 触发时机（自动，无需用户要求）

**Code Review 完成后，必须自动执行**本 skill，无需用户再次请求。包括：

- code-reviewer 子任务返回后
- 人工完成代码评审后
- 用户请求「代码 review」且评审结论已输出后

其他触发：用户明确要求「测试功能 review」「测试用例设计」时。

---

## 工作流（四步）

```
1. 意图锚定 → 2. 数据流审计 → 3. 黑盒用例设计 → 4. 代码对照审计 → 输出结论
```

---

## Step 1: 意图锚定（Intent Anchoring）

**目的**：明确"这次改动要达成什么效果"，作为测试用例的锚点。

### 按任务类型自适应

不是所有改动都有 PRD。根据任务类型选择对应的锚定方式：

#### 类型 A: 新功能（有 PRD / 有需求描述）

1. 定位需求源：`docs/features/*/prd.md`、commit message、用户指令
2. 提取功能清单 → 输出**需求追踪矩阵**

| 功能点 | 需求描述 | 代码实现位置 | 状态 |
|--------|----------|-------------|------|
| F-1 | xxx | `file.go:L42` | ✅ 已实现 / ❌ 缺失 / ⚠️ 偏差 |

3. 检查：遗漏（需求有但代码没有）、过度实现（代码有但需求没有）、语义偏差

#### 类型 B: Bug 修复

1. 明确 bug 三要素：

| 项 | 内容 |
|----|------|
| **现象** | 实际发生了什么（如：`FileSize` 入库为 0） |
| **预期** | 应该是什么（如：`FileSize` 应为文件实际大小） |
| **影响范围** | 哪些功能/用户受到影响 |

2. 确认修复意图：这个 fix 要解决的是什么？是否有**同类问题**需要一并排查？
3. 圈定**回归范围**：这个 fix 触及了哪些模块？这些模块的其他功能是否可能被影响？

#### 类型 C: 重构 / 编码规范修复

1. 明确**行为等价约束**：重构前后，对外暴露的行为必须完全一致
2. 列出**不变量清单**：

| 不变量 | 验证方式 |
|--------|---------|
| API 输入输出不变 | 对比请求/响应 schema |
| 错误码不变 | 对比错误返回 |
| 副作用不变 | DB 写入、消息发送、配额变化 |

3. 重构范围越大，不变量清单越要详细

#### 类型 D: 仅配置/文档变更

- 快速确认：变更是否引入了行为变化？如果没有，输出"仅配置/文档变更，无需功能测试"
- 如果配置变更影响运行时行为（如限流阈值、开关），仍需设计用例

### 输出

无论哪种类型，Step 1 必须输出一段**简明的意图描述**（3-5 句话），作为后续所有用例设计的锚点。格式：

```
**任务类型**：新功能 / Bug修复 / 重构 / 配置变更
**意图**：[这次改动要达成什么效果]
**范围**：[涉及哪些模块/文件]
**回归关注**：[可能被波及的相邻功能]
```

## Step 2: 数据流审计（Data Flow Audit）

**目的**：追踪数据从入口到出口/存储的完整路径，发现字段丢失、硬编码、默认值滥用等问题。

> **这是抓住 `FileSize: 0` 类 bug 的核心步骤。**

### 2.1 Struct 构造审计（逐字段）

对本次改动中**每一个 struct 字面量构造**（如 `&XxxRequest{...}`），逐字段检查：

| 字段 | 赋值来源 | 值是否合理 | 风险 |
|------|---------|-----------|------|
| FieldA | `param.X` | ✅ 来自上游 | - |
| FieldB | `0`（硬编码） | ❌ | **Critical: 业务上不应为零** |

#### 审计规则

**R-DATA-1: 硬编码零值嗅探**
- 对 struct 中每个字段，如果值为字面量 `0`、`""`、`nil`、`false`，必须追问：
  - 这个零值在业务上是否有意义？
  - 是否应该从参数/上游获取真实值？
  - 是"暂时不知道所以先写 0"还是"确实为零"？
- 例外：`CreatedAt: time.Now()` 这种明确的初始化不算硬编码零值
- 例外：`Status: StatusPending` 等明确的初始状态常量
- **典型案例**：`FileSize: 0` — AI 生成的文件必然有大小，0 会导致配额计算错误

**R-DATA-2: 字段遗漏检测**
- 对比 struct 定义的所有字段 vs 构造时赋值的字段
- 未赋值的字段 = Go 零值，追问：这个零值是否安全？
- 特别关注：`ID`、`Status`、`CreatedAt`、`UserID` 等不应为零的字段

**R-DATA-3: 数据源可信性**
- 来自用户输入 → 是否经过校验/清洗？
- 来自外部 API → 是否处理了空响应？
- 来自配置 → 是否有默认值兜底？

### 2.2 数据链路追踪

对关键数据，追踪 `入口 → 转换 → 存储/输出` 的完整路径：

```
[入口] → 解析 → 处理 → 存储/输出
  ↑        ↑       ↑        ↑
 完整？  映射对？  有损？  字段匹配？
```

检查点：
- **字段映射**：`request.FileName` → `model.FileName` 映射是否正确？
- **类型转换**：`string → int64` 是否有溢出/格式错误处理？
- **单位转换**：字节/KB/MB 是否一致？时间戳秒/毫秒是否一致？
- **空值传播**：上游为 nil 时下游是否会 panic？

## Step 3: 黑盒用例设计（Black-Box Test Design）

> **黑盒视角**：假设你不知道代码怎么写的，只知道"这个接口/功能的输入和输出是什么"，从外部行为的角度设计用例。

### 用例表结构

| 用例ID | 维度 | 场景 | 前置条件 | 输入/步骤 | 预期结果 | 优先级 |
|--------|------|------|----------|----------|----------|--------|
| T-1 | 正常流 | ... | ... | ... | ... | P0 |

### 维度 A: 正常流（Happy Path）— P0

核心业务场景，必须 100% 覆盖：
- 最典型的使用场景
- 所有必填参数都正确的情况
- 预期的完整执行路径

### 维度 B: 字段级校验（Field Validation）— P0

**对每个输入字段逐一设计用例**，这是抓 bug 的密集区：

| 字段类型 | 必测场景 |
|---------|---------|
| string | 空串 `""`、纯空格 `"  "`、超长（>max）、特殊字符、前后空格 |
| int64/int | `0`、负数、`MaxInt64`、边界值（如 `PageSize=0`） |
| *int64（可选） | `nil` vs `&0` 的行为差异 |
| []T（切片） | `nil`、空切片 `[]T{}`、单元素、超大切片 |
| enum/status | 每个有效值 + 一个无效值 |
| URL | 空、无协议、非法字符、超长 |
| 时间 | 零值、过去时间、未来时间 |
| 文件相关 | `FileSize=0`、`FileSize<0`、MIME 与扩展名不匹配 |
| ID 类 | `0`、负数、不存在的 ID、已删除的 ID、他人的 ID（越权） |

### 维度 C: 异常与错误路径（Error Path）— P0

- 依赖服务失败（DB 宕机、Redis 超时、外部 API 不可用）
- 资源不存在（ID 无效、已被删除）
- 权限不足（非 owner 操作他人资源）
- 事务回滚（中间步骤失败，前面的步骤是否回滚？）
- 并发冲突（同时操作同一资源）

### 维度 D: 组合测试（Combination）— P0

> **组合测试是黑盒测试的核心**，不是可选项。
> 任何功能都不是孤立存在的，必须考虑参数组合、状态组合、操作序列。

#### D.1 参数组合（输入维度交叉）

当接口有 2+ 个输入参数时，必须考虑参数之间的交叉影响：

**方法：等价类 × 因子表**

1. 对每个参数划分等价类（有效值/边界值/无效值）
2. 用 pairwise（两两组合）生成最小覆盖集
3. 重点关注：**一个参数为边界值时，其他参数的行为是否受影响**

示例（创建素材接口）：

| 组合ID | FileName | FileSize | MimeType | FolderID | 预期 |
|--------|----------|----------|----------|----------|------|
| C-1 | 正常 | 正常 | 正常 | 存在 | ✅ 成功 |
| C-2 | 空串 | 正常 | 正常 | 存在 | ❌ 文件名校验失败 |
| C-3 | 正常 | 0 | 正常 | 存在 | ❌ 文件大小无效 |
| C-4 | 正常 | 正常 | 无效MIME | 存在 | ❌ MIME校验失败 |
| C-5 | 正常 | 正常 | 正常 | 不存在 | ❌ 文件夹不存在 |
| C-6 | 超长 | 负数 | 空 | 0 | ❌ 多参数同时非法 |

#### D.2 状态组合（资源当前状态 × 操作）

**方法：状态-操作矩阵**

列出资源的所有可能状态，检查每个操作在每个状态下的行为：

| 操作 \ 状态 | 正常 | 已删除 | 已锁定 | 过期 | 不存在 |
|-------------|------|--------|--------|------|--------|
| 查看 | ✅ | ? | ? | ? | ? |
| 编辑 | ✅ | ? | ? | ? | ? |
| 删除 | ✅ | ? | ? | ? | ? |

每个 `?` 都是一个用例——确认行为是否正确处理，还是会产生意外结果。

#### D.3 操作序列（时序组合）

考虑操作的先后顺序和重复性：

- **链式操作**：创建 → 更新 → 查询 → 删除，中间状态是否一致？
- **重复操作**：同一操作执行两次的行为（幂等性）
- **逆序操作**：先删除再创建，是否正常？
- **并发操作**：两个请求同时修改同一资源

#### D.4 跨功能组合（改动波及面）

> **修 bug 或实现功能时最容易忽略的维度**：你的改动对相邻功能有什么影响？

- 修改了 service A 的逻辑，有哪些 handler/其他 service 在调用 A？
- 改了数据库表结构，有哪些查询会受影响？
- 改了 DTO 字段，前端是否还兼容？

**必须列出受影响的调用方，并为每个调用方设计至少 1 个回归用例。**

### 维度 E: 边界与极值（Boundary）— P1

- 分页：`page=0`、`page=1`、`page=MaxInt`、`pageSize=0`
- 列表为空时的行为
- 恰好等于限制值（如配额刚好用完）
- 时间边界（跨天、跨月）

### 用例设计原则

1. **黑盒优先**：先从外部行为设计用例，再对照代码验证
2. **组合必做**：有 2+ 输入参数的接口，必须做参数组合；有状态的资源，必须做状态矩阵
3. **一个用例一个断言**：每个用例只验证一个行为
4. **回归意识**：bug 修复和功能开发都要考虑"我的改动会不会破坏别的东西"

## Step 4: 代码对照审计

逐条对照测试用例，**阅读实际代码**检查实现。

### 4.1 逐用例对照

对每个 P0 用例，按以下模板检查：

```
用例 T-X: [场景描述]
├── 代码位置: file.go:L42-L56
├── 输入处理: ✅ 有校验 / ❌ 无校验 / ⚠️ 校验不完整
├── 逻辑正确: ✅ / ❌ [具体问题]
├── 错误处理: ✅ 有错误返回 / ❌ 静默忽略 / ⚠️ 错误信息不明确
├── 日志记录: ✅ / ❌ / N/A
└── 判定: PASS / FAIL / WARN
```

### 4.2 重点审计清单

| # | 审计项 | 检查方法 | 典型 bug |
|---|--------|---------|---------|
| 1 | **硬编码零值** | 搜索 struct 构造中的 `0`、`""`、`nil` | `FileSize: 0` 应取实际值 |
| 2 | **未赋值字段** | 对比 struct 定义与构造赋值 | 遗漏关键字段导致零值入库 |
| 3 | **入参未校验** | 检查 handler/service 入口是否有 validate | 非法值穿透到 DB |
| 4 | **错误吞没** | 搜索 `_ =` 和空 `if err != nil` | 错误被静默忽略 |
| 5 | **nil 解引用** | 检查指针/可选值使用前是否判 nil | panic 崩溃 |
| 6 | **越权访问** | 检查是否校验 UserID 归属 | 访问他人数据 |
| 7 | **事务完整性** | 多步操作是否在同一事务中 | 部分成功部分失败 |
| 8 | **幂等性** | 重复调用是否安全 | 重复创建/扣费 |
| 9 | **返回值完整性** | 返回的 DTO 是否包含所有必要字段 | 前端拿不到关键数据 |
| 10 | **回归安全** | 改动是否影响了其他调用方 | 修了 A 结果 B 坏了 |

### 4.3 副作用审计

- 创建操作：是否正确更新了配额/计数？
- 删除操作：关联数据是否清理？
- 更新操作：`updated_at` 是否更新？
- 调用链一致性：handler → service → repository 的参数传递和错误映射是否正确？

## 输出结论

### 格式

```markdown
# 意图锚定

**任务类型**：xxx
**意图**：xxx
**范围**：xxx
**回归关注**：xxx

# 数据流审计发现

[Step 2 中发现的字段/数据问题，如无则写"无异常"]

# 测试用例表

[Step 3 的完整用例表，含组合测试用例]

# 评审结论

## 需修复（Critical）
- [问题描述]：代码位置 + 逻辑缺陷 + 影响范围
  - **根因**：为什么会出现
  - **修复建议**：具体修改方向

## 已修复
- [在本次 review 中发现并已修复的问题]

## 可选优化（Suggestion）
- [非必须但可提升健壮性的优化点]

## 结论
- 用例总数：X 条（含组合用例 Y 条）
- 通过：X 条
- 需修复：X 条（Critical）
- 可选优化：X 条
- **总体判定**：✅ 可提交 / ❌ 需修复后重审
```

### 问题分级

| 级别 | 定义 | 典型场景 | 处理 |
|------|------|---------|------|
| Critical | 导致功能错误或数据损坏 | `FileSize=0` 入库、越权访问、事务不完整、组合参数下崩溃 | **必须修** |
| Warning | 功能正确但存在隐患 | 缺少入参校验、错误信息不明确、缺少回归测试覆盖 | 强烈建议修 |
| Suggestion | 可提升健壮性或可维护性 | 添加注释、优化错误提示文案 | 视情况 |
| OK | 用例完全满足 | - | 无需改动 |

执行本 SKILL 后，必须确认以下项全部覆盖：

```
□ Step 1: 意图描述已输出（含任务类型、意图、范围、回归关注）
□ Step 2: 每个 struct 构造已逐字段审计，无硬编码零值遗漏
□ Step 2: 数据从入口到存储的链路已追踪
□ Step 3: 四个核心维度（正常流/字段校验/异常/组合）均有用例
□ Step 3: 有 2+ 参数的接口已做参数组合用例
□ Step 3: 有状态的资源已做状态-操作矩阵
□ Step 3: Bug修复/功能开发已列出受影响的调用方和回归用例
□ Step 4: 每个 P0 用例已对照代码确认
□ 结论中包含用例统计（含组合用例数量）和总体判定
```

## 输出原则

1. **黑盒优先**：先不看代码，从输入/输出行为设计用例；再用用例审计代码
2. **字段穷举**：struct 构造中的每个字段都要追问"这个值从哪来？合理吗？"
3. **零值即嫌疑**：看到 `0`、`""`、`nil` 就要追问，直到确认是有意为之
4. **组合必做**：多参数接口和有状态资源，组合测试是 P0 不是 P1
5. **回归必想**：每次改动都问"我的改动还会影响谁？"
6. **场景自适应**：新功能查 PRD、bug 修复查现象/预期/影响、重构查行为等价
7. **只审本次改动**：聚焦与本次功能相关的逻辑，不无限扩散
8. **给出可操作建议**：指出具体位置、根因和修改方向
