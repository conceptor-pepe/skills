# Improve Codebase Architecture（架构改进探索）

参考 John Ousterhout《A Philosophy of Software Design》中的深层模块理念：
**深层模块** = 接口简单 + 隐藏大量实现。深层模块更易测试、更易被 AI 导航，可以在边界处测试而非内部。

## 流程

### 第 1 步：自然探索代码库

使用 explore 子代理有机地探索代码库，**不要使用僵化的规则**，而是记录你实际遇到的摩擦点：

- 理解一个概念需要在很多小文件之间跳来跳去？
- 某个模块的接口复杂度几乎等于实现复杂度（浅层模块）？
- 为了可测试性而提取了纯函数，但真正的 bug 藏在调用方式里？
- 紧耦合的模块导致改动一处、影响多处？
- 哪些代码难以测试，或根本没有测试覆盖？

**遇到摩擦就是信号。**

### 第 2 步：列出候选改进点

以编号列表呈现候选项，每条包含：

- **涉及模块**：哪些模块/概念相关
- **耦合原因**：共享类型、调用模式、概念共同所有权
- **测试影响**：改进后可替代哪些现有测试，或新增哪些边界测试

此时不要提出接口设计方案。先问用户：**「你想深入探讨哪个？」**

### 第 3 步：用户选择候选项

### 第 4 步：描述问题空间

在启动并行子代理之前，向用户说明：

- 新接口需要满足哪些约束
- 它需要依赖哪些现有组件
- 一段示意性伪代码（不是方案，只是让约束更具体）

说明完毕后，**立即进入第 5 步**（不等用户回复，让他们在子代理工作时思考）。

### 第 5 步：并行产出 3 个接口设计

使用 Task 工具同时启动 3 个子代理，每个代理产出**截然不同**的接口设计。

每个子代理的技术简报包含：相关文件路径、耦合细节、需要隐藏的复杂性，以及各自的设计约束：

- **子代理 A**：「最小化接口 — 目标 1-3 个入口点」
- **子代理 B**：「最大化灵活性 — 支持多种使用场景和扩展」
- **子代理 C**：「优化最常见的调用方 — 让默认用法最简单」

每个子代理输出：
1. 接口签名（类型、方法、参数）
2. 调用示例（调用方如何使用）
3. 内部隐藏了什么复杂性
4. 依赖策略（依赖如何管理）
5. 取舍分析

### 第 6 步：比较并给出推荐

依次呈现 3 个设计，然后用散文段落对比它们。

**必须给出明确推荐**：你认为哪个设计最强，为什么。如果不同设计的元素可以组合，提出混合方案。不要只给菜单让用户选——用户需要明确的观点。

### 第 7 步：生成 RFC 文档

用户确认方向后（或采纳推荐），生成 RFC 文档保存至 `docs/prds/YYYY-MM-DD-arch-<name>.md`。

## RFC 文档模板

```markdown
# 架构 RFC：<改进名称>

> 创建日期：YYYY-MM-DD
> 状态：提案

## 背景与问题

描述当前架构的摩擦点，以及为什么需要改进。

## 涉及模块

- 模块 A：当前状态和问题
- 模块 B：耦合关系

## 提议的接口设计

### 接口签名

```go
// 示意性代码，非最终实现
```

### 调用示例

展示改进后的典型用法。

### 隐藏的复杂性

这个接口背后封装了哪些复杂逻辑。

## 备选方案

简述被否定的其他设计及原因。

## 测试策略

改进后如何在边界处测试，替代哪些现有测试。

## 迁移策略

如何从当前实现过渡到新接口，是否需要分阶段。

## 风险与取舍

主要风险点和已知取舍。
```

---

## 注意事项

- 探索阶段要有机进行，不要套模板
- 并行子代理的设计必须**截然不同**，不要三个都是同一思路的微小变体
- 推荐必须有理由，不能模棱两可
- RFC 专注"接口设计"，不是完整实施计划
- 用中文输出所有内容
