# Xiezhi Workflow / 獬豸工作流（强制）

> Xiezhi (獬豸), the beast that discerns right from wrong. 能辨曲直、触不直。本工作流以其把关之能，约束 Go 开发全流程。

编写或修改 Go 代码时，**必须**按以下流程执行，禁止跳过或颠倒顺序。

## 执行顺序（流水线）

```
0. 功能启动 ──► 0.5 方案简报 ──► 1. 编写 ──► 2. 编码审计 ──► 3. [模块审计] ──► 4. Code Review ──► 5. 测试评审 ──► 6. 提交总结
```

## 各阶段职责

| 阶段 | 职责 | 执行内容 | 对应规则 |
|------|------|----------|----------|
| 0. 功能启动 | PRD + 上下文固化 | 检查 PRD 存在；创建/读取 task_plan.md | `feature-start-workflow.md` |
| 0.5 方案简报 | 方案对齐 | 输出架构影响+API契约+设计决策，用户确认后动手 | `task-design-brief.md` |
| 1. 编写 | 遵守编码规范 | 遵循 go-code-standards，输出前自查 | `go-code-standards.md` |
| 2. 编码审计 | 自动检查 | 执行 `audit.py`，不通过则修复 | `go-audit-integration.md` |
| 3. 模块审计 | 按 Gate 审计 | 模块级任务时按 Gate 0→4 执行 | `module-review-gates.md` |
| 4. Code Review | 代码评审 | 人工或 code-reviewer 子任务 | — |
| 5. 测试评审 | 测试用例评审 | Code Review 完成后**立即**执行 | `test-functionality-review.md` |
| 6. 提交总结 | 输出变更总结 | 按 commit-summary 格式输出 | `commit-summary.md` |


## 阶段 0：功能启动（新功能必须）

- 触发：用户意图为「新功能」「开发 xxx」「实现 xxx」或任务步骤 > 3
- 执行 `feature-start-workflow.md` 的完整流程（PRD 检查 → task_plan.md 检查）
- 禁止：未完成阶段 0 就进入阶段 0.5

## 阶段 0.5：方案简报（编码前必须）

- 触发：每个 Task 开始编码前（有接口/跨模块/新增模块的改动）
- 执行 `task-design-brief.md`，输出简报（架构影响 + API 契约 + 设计决策 + 待确认）
- **用户确认后**才可进入阶段 1
- 禁止：未输出简报就开始写代码
- 禁止：未等用户确认就自行开始编码
- 例外：纯 bugfix、审计修复、文档修改可跳过

## 阶段 1：编写

- 编写前遵守 `go-code-standards.md` 全部规范
- 禁止：无注释公开函数、未处理错误、超 50 行函数、阻塞操作不传 context

## 阶段 2：编码审计（强制）

- 命令：`python3 ~/.kiro/skills/go-audit/scripts/audit.py <本次修改的.go文件>`
- 审计不通过 → **先修复**，再重跑，直到通过
- 禁止：跳过审计、手动声称通过、用其他方式替代

## 阶段 3：模块审计（适用时）

- 对模块做审计/重构时，按 `module-review-gates.md` 执行 Gate 0→4
- 前一 Gate 未通过不得进入下一 Gate

## 阶段 4→5：Code Review 后测试评审（强制）

- Code Review 完成后，**必须立即**执行测试功能 Review
- **执行方式**：按 `test-functionality-review.md` 三步流程执行（用例设计 → 按用例评审 → 输出结论）
- **输出要求**：必须在当前回复中输出完整的测试用例表和评审结论
- 禁止：等待用户说「测试功能 review」再执行
- 禁止：询问「是否进行测试功能 review？」
- 禁止：以「无业务逻辑」「仅数据层」「仅 Repository」为由跳过

## 阶段 5→6 门禁（硬约束）

**阶段 6 的前置条件是阶段 5 已完成。** 违反此约束时：

1. 检查当前会话中是否存在测试用例表和「评审结论」段落
2. 如果**不存在** → **立即停止**，先执行阶段 5，完成后再进入阶段 6
3. 如果**存在** → 在提交总结中填写「测试评审」必填段

## 阶段 6：提交总结（强制）

- 任务/Commit 完成后，按 `commit-summary.md` 格式输出
- **必须**包含「测试评审」必填段（来自阶段 5 的结论）
- 禁止：总结中包含催促提交用语（如「请审计后提交」）

## 任务收尾确认清单（强制输出）

每次任务结束前，**必须显式输出以下清单并逐项确认**：

```
## 任务收尾检查 ✅
- [x/□] 阶段0.5：方案简报已输出并获用户确认（如适用）
- [x/□] 阶段2：audit.py 已执行，结果 ___
- [x/□] 阶段4：Code Review 已完成
- [x/□] 阶段5：测试评审已完成
- [x/□] 阶段6：提交总结已输出
- [x/□] task_plan.md 当前 Phase 已标记 complete（如有）
- [x/□] docs/api-index.md 已更新（如有新增/修改接口）
```

## 规则优先级

1. 本工作流（执行顺序不可违反）
2. 各阶段对应规则
3. 用户临时指令
