# Write a PRD（产品需求文档）

适用于：用户说「写 PRD」「产品需求」「新功能规划」「需求文档」，或在开始实施前需要明确需求边界时。

## 流程

### 第 1 步：深度访谈

逐一提问（每次只问一个问题），围绕以下维度展开：

- **问题**：用户当前面临什么痛点？现状是什么？
- **目标用户**：谁会使用这个功能？有哪些角色？
- **解决方案**：有哪些可能的方案？用户倾向哪个方向？
- **约束**：有哪些技术限制、时间限制、业务规则？
- **成功标准**：如何衡量这个功能做好了？

优先使用多选题提问，避免开放式问题让用户无从下手。

### 第 2 步：探索代码库

访谈结束后，探索代码库验证用户的判断：

- 当前已有哪些相关模块？
- 哪些接口/数据模型需要改动？
- 是否存在现有代码可以复用？

### 第 3 步：确认模块边界

向用户呈现本次改动涉及的主要模块（Schema / Repository / Service / Handler），确认模块划分是否符合预期。

### 第 4 步：输出 PRD

使用以下模板写 PRD，保存到 `docs/features/<feature-name>/prd.md`。

---

## PRD 模板

```markdown
# PRD: <功能名称>

> 创建日期：YYYY-MM-DD
> 状态：草稿 / 评审中 / 已确认

## 问题陈述

从用户视角描述当前面临的问题和痛点。

## 解决方案

从用户视角描述解决方案，不涉及具体实现细节。

## 用户故事

1. 作为 <角色>，我希望 <功能>，以便 <收益>。
2. ...

## 实施决策

- 涉及的模块（新建 / 改造）
- API 设计（路由、请求/响应结构）
- 数据库 Schema 变更
- 认证/鉴权策略

## 测试决策

- 需要补充测试的模块
- 测试参考（代码库中已有的类似测试）

## 不在范围内

明确说明本次不做什么。
```

---

## 衔接下一步

PRD 保存后，提示用户：

> 下一步选项：
> 1. **拆分实施计划** → 使用 `prd-to-plan` steering，将 PRD 拆解为垂直切片计划
> 2. **直接实施** → 遵循 `xiezhi-workflow.md`，生成详细实施步骤

## 注意事项

- 每次只问一个问题，不要一次抛出多个问题
- 能通过探索代码库回答的问题，先探索再问
- PRD 专注"做什么"和"为什么"，不描述"怎么做"
- 用中文输出所有内容
