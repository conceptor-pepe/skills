# Task Design Brief / 任务方案简报（强制）

> **核心原则**：先对齐方案再写代码。设计文档是契约不是参考，最简方案优先。

## 触发条件

**每个 Task 开始编码前**，必须先输出方案简报并等待用户确认。

触发场景：
- task_plan.md 中的每个 Phase/Task 开始实现时
- 用户说「实现 xxx」「开始 xxx」时
- 任何涉及 **新增接口、新增模块、跨模块调用** 的改动

不触发：
- 纯 bugfix（改 1-2 行修复已知问题）
- 纯格式/注释/文档修改
- 审计修复（audit 不通过的修复）

## 简报模板（强制格式）

```markdown
## Task Brief: [task 名称]

### 架构影响

**模块依赖（本次新增/修改的跨模块调用）**
// 用树形图展示，只列新增或变更的依赖

**耦合点**
| 耦合 | 说明 | 风险 |
|------|------|------|

**模块内部新增/变更**
// 列出新增和变更的文件，一行一个，标明职责

### API 契约（对照设计文档）

| 接口 | 路由 | 关键响应字段 |
|------|------|-------------|

> ⚠️ 字段名必须从设计文档原文摘取，不可自行命名。

### 设计决策（仅列有选择的点）

| 决策 | 方案 | 理由 |
|------|------|------|

### 待确认

1. [需要用户拍板的问题]
```

## 执行规则

1. **简报先行**：未输出简报或用户未确认前，**禁止写任何业务代码**
2. **设计文档是契约**：API 字段名、响应结构严格按设计文档，偏差必须显式提出
3. **最简方案优先（YAGNI）**：先问"不加这个行不行"
4. **等待确认**：输出简报后等待用户回复，不可自行假设确认

### 禁止

- **禁止**跳过简报直接写代码
- **禁止**简报中遗漏 API 契约段（有接口变更时）
- **禁止**自行发明字段名/路由名而不对照设计文档
- **禁止**在用户未确认的情况下开始编码

### 粒度控制

整个简报控制在 **30-50 行**以内，用户 1-2 分钟可扫完。只列有分歧的决策点，显而易见的实现（CRUD、标准分层）不列。

## 与工作流的衔接

```
0. 功能启动 ──► 0.5 方案简报（本规则） ──► 1. 编写 ──► ...
```

阶段 0.5 未通过（用户未确认或要求修改）时，禁止进入阶段 1。
