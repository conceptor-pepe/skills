# 接收代码 Review（Receiving Code Review）

## 核心原则

**先验证，再实现。先理解，再动手。技术正确性优先于社交舒适度。**

---

## 处理流程

```
收到 review 反馈时：

1. READ       完整读完所有反馈，不要边读边改
2. UNDERSTAND  用自己的话复述需求（不确定就问）
3. VERIFY     对照 codebase 验证反馈是否正确
4. EVALUATE   这个建议适合我们的项目吗？
5. RESPOND    技术性确认，或有理由地反驳
6. IMPLEMENT  逐条实现，每条改完后验证
```

---

## 禁止的回应方式

**不要说：**
- 「你说得对！」（表演性同意）
- 「好建议！」「很棒的反馈！」
- 「我马上改」（在验证之前）

**应该：**
- 复述技术需求
- 提问澄清
- 有技术依据地反驳
- 直接动手改（行动胜于表态）

---

## 反馈不清楚时

```
如果有任何一条反馈不清楚：
  先停下来，不要实现任何东西
  问清楚再动手

原因：各条反馈可能相互关联，部分理解 = 错误实现
```

示例：
```
收到「修复第 1-6 条」，理解了 1、2、3、6，不确定 4、5

❌ 错误：先改 1、2、3、6，再问 4、5
✅ 正确：「我理解 1、2、3、6，请说明 4 和 5 再开始」
```

---

## 何时反驳

以下情况应该反驳，而不是盲目实现：

- 建议会破坏现有功能
- 建议违反 YAGNI（添加没有调用方的功能）
- reviewer 不了解完整上下文
- 技术上对我们的栈不适用
- 与已有架构决策冲突

**如何反驳：**
- 用技术理由，不要情绪化
- 引用现有代码或测试作为依据
- 如果涉及架构决策，先和团队确认

**YAGNI 检查：**
```
如果 reviewer 建议「完善某个功能」：
  先搜索 codebase，确认是否有调用方

  没有调用方 → 「这个接口目前没有被调用，是否需要？」
  有调用方   → 正常实现
```

---

## 实现顺序

收到多条反馈时：

1. 先澄清所有不清楚的条目
2. 按优先级实现：
   - 阻塞性问题（编译失败、安全漏洞）
   - 简单修复（typo、import）
   - 复杂修复（重构、逻辑变更）
3. 每条改完后单独验证（`go build ./...` / `go test`）
4. 确认没有引入回归

---

## 确认正确反馈的方式

当反馈确实正确时：

```
✅ 「已修复。[简短说明改了什么]」
✅ 「发现了 [具体问题]，已在 [位置] 修复」
✅ 直接改代码，用 diff 说话

❌ 「你说得对！」
❌ 「感谢指出！」
❌ 任何感谢表达
```

行动本身就是最好的回应。

---

## 反驳后发现自己错了

```
✅ 「验证了一下，你是对的。我之前理解有误，原因是 [X]。正在修复。」

❌ 长篇道歉
❌ 解释为什么当时反驳
❌ 过度解释
```

说清楚，继续干活。
