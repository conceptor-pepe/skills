# 系统化调试（Systematic Debugging）

## 铁律

```
没有找到根因之前，禁止提出任何修复方案。
```

「先试试改这里」= 失败。症状修复不是修复。

---

## 四个阶段（必须按顺序完成）

### Phase 1: 根因调查

**在做任何修改之前，必须完成以下步骤：**

**1. 完整阅读错误信息**
- 不要跳过 warning 和 stack trace
- 记录：行号、文件路径、错误码
- 错误信息通常已经包含答案

**2. 稳定复现**
- 找到最小可复现路径
- 确认是否每次都出现
- 无法稳定复现 → 先收集更多数据，不要猜

**3. 检查最近变更**
- `git diff`、最近 commit
- 新增依赖、配置变更、环境差异

**4. 多组件系统：加诊断日志定位边界**

当系统有多个组件（handler → service → repository → DB）时，**在提出修复前**，先在每个组件边界加日志：

```go
// 每个边界记录：入参、出参、状态
logger.Info("进入 service", zap.Any("req", req))
// ... 执行逻辑
logger.Info("离开 service", zap.Any("result", result), zap.Error(err))
```

运行一次，看日志确认**哪一层**出了问题，再针对那一层调查。

**5. 向上追溯数据流（根因追踪）**

当错误出现在调用栈深处时，**不要在症状点修复**，要向上追溯：

```
错误出现在哪里？
  → 是谁调用了这里？传入了什么值？
    → 是谁调用了上一层？
      → 继续向上，直到找到错误值的来源
        → 在源头修复
```

示例：
```
repository.Query() 收到空的 userID
  → service.GetUser() 传入了空值
    → handler 从 ctx 取值时已经是空
      → middleware 没有正确注入 userID ← 根因在这里
```

---

### Phase 2: 模式分析

找到根因后，分析模式再修复：

- 在同一个 codebase 里找**类似的正常工作的代码**
- 对比：正常的和异常的有什么不同？
- 理解依赖：这段代码依赖哪些配置、环境、前置条件？

---

### Phase 3: 提出假设并验证

- 每次只改一个变量
- 改之前写下预期：「如果假设正确，改完后应该看到 X」
- 用日志/测试确认，不靠推断下结论

---

### Phase 4: 修复并防御

**修复根因，同时加防御层：**

- 在根因处修复
- 在每个关键边界加校验（不要只修一处）
- 加测试覆盖这个 case，防止回归
- 检查同类边界是否有相同漏洞

```go
// 示例：多层防御
// Layer 1: handler 层校验入参
if req.UserID == 0 {
    return response.BadRequest(c, "user_id 不能为空")
}

// Layer 2: service 层二次校验
if params.UserID == 0 {
    return fmt.Errorf("GetUser: userID 不能为零值")
}
```

---

## 特别提醒：越紧急越要走流程

- 时间紧 → 更容易乱猜 → 更容易引入新 bug
- 「就改这一行」的直觉修复，往往掩盖真正的问题
- 系统化调试比反复试错更快

---

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 没有假设就随机改代码 | 先完成 Phase 1 再动手 |
| 在症状点修复 | 向上追溯到根因 |
| 「改好了但不知道为什么」 | 没找到根因，问题会复发 |
| 修复后不加测试 | 必须加回归测试 |
| 多组件系统靠猜哪层出问题 | 加诊断日志，用证据定位 |
