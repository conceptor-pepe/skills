# Go 代码完整审查流程

## 使用场景

- 用户要求「完整代码审查」「代码 review」
- 提交代码前的最终检查
- 完成功能开发后的质量验证

## 执行流程（三步）

### Step 1: 编码规范审计

运行 audit.py 脚本检查所有修改的 Go 文件：

```bash
python3 ~/.cursor/skills/go-audit/scripts/audit.py --dir <修改的目录>
```

或针对具体文件：

```bash
python3 ~/.cursor/skills/go-audit/scripts/audit.py <file1.go> <file2.go>
```

**检查项（9 项）**：
1. 公开函数注释
2. 函数长度（≤50 行）
3. 函数参数（≤5 个）
4. 嵌套深度（≤2 层）
5. 错误处理
6. 错误分支日志
7. 无反射
8. context 传递
9. 接口最小化

**输出**：
- ✅ 全部通过：`编码规范审计 ✅ (9/9)`
- ❌ 有违规：列出违规项，**必须先修复**

---

### Step 2: 测试功能 Review

从测试角度设计用例并评审代码。

#### 2.1 设计测试用例

| 用例ID | 场景 | 前置条件 | 步骤 | 预期结果 |
|--------|------|----------|------|----------|
| X-1 | ... | ... | ... | ... |

**覆盖维度**：
- 正常流：Happy path
- 边界：空值、极值、零值
- 异常：错误输入、依赖失败
- 集成：跨模块交互

#### 2.2 按用例评审代码

检查点：
- 输入是否 trim/校验
- 边界长度是否有校验
- 错误是否正确包装
- JOIN/关联字段是否一致
- 脱敏逻辑的边界处理

#### 2.3 输出结论

```markdown
# 测试用例
[用例表]

# 评审结论

## 需修复（Critical）
- [问题描述]：代码位置/逻辑缺陷

## 可选优化（Suggestion）
- [优化点]：非必须但可提升健壮性

## 结论
- 功能是否 OK
- 建议修复项
```

---

### Step 3: 输出最终报告

汇总两个阶段的结果：

```markdown
# Go 代码完整审查报告

## 1. 编码规范审计
编码规范审计 ✅ (9/9)

## 2. 测试功能 Review
[测试用例表]

### 评审结论
- 需修复：X 项
- 可选优化：Y 项
- 整体评价：[OK/需修复]

## 建议
[具体修复建议]
```

---

## 执行原则

1. **顺序执行**：先编码规范审计，通过后再做测试 review
2. **必须修复**：编码规范审计不通过时，必须先修复
3. **聚焦改动**：只审查本次修改的代码
4. **可操作建议**：给出具体位置和修改方向
